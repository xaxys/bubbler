# Bubbler E2E — Unified Test Spec

Single source of truth for every per-language e2e test driver
(`tests/c/main.c`, `tests/cpp/main.cpp`, `tests/go/<pkg>/<pkg>_test.go`,
`tests/java/Main.java`, `tests/python/test_main.py`,
`tests/csharp/Program.cs`, `tests/cjs/test.mjs`, `tests/esm/test.mjs`).

`run_tests.sh` regenerates each driver from this Go spec before compiling, so
adding a test case is a one-place edit — never seven.

## Running the generator

```sh
go run ./e2e/spec -lang=<lang> -out=<path>
```

Languages: `c`, `cpp`, `go`, `java`, `python`, `csharp`, `cjs`, `esm`.

For `-lang=go`, `-out` is the per-language root directory (it writes
`<out>/<package>/<package>_test.go` per `.bb` package). For every other
language it is the single output file.

## Adding a new struct

1. Add the struct to `e2e/testcase.bb` (or another `.bb` already
   generated by `run_tests.sh`).
2. Add a corresponding scenario function to `scenarios.go` and register it in
   `AllScenarios()`. The scenario is language-neutral data:
   - `Setup`: list of `FieldVal`s assigning every field on the input struct.
   - `Assert` (optional): list of `AssertField`s to check after decode. If
     empty, the framework asserts each `Setup` value matches.
   - `Errors` (optional): byte-patches applied to the encoded buffer that
     must cause `decode` to return the given negative value.
   - `DecodeSizeChecks` (dynamic structs only): byte-array probes for
     `decode_size`, plus optional source-case truncation tests.
3. Run `bash e2e/run_tests.sh` (or `make e2e`) to regenerate every driver.

## Adding a new case to an existing struct

Append a `Case{}` to that struct's `Scenario.Cases`. Default tolerance for
floats is **bit-exact** (no FP epsilon). Set `Tol > 0` on individual asserts
to switch to near-equality.

## Float corner cases

`scenarios.go::floatSpecials()` covers `+/-Inf`, quiet NaN (positive and
negative sign), `-0.0`, `+0.0`, `FLT_MAX/DBL_MAX`, smallest normal, smallest
subnormal — round-tripped through encode/decode and compared at the bit
level. Each language driver renders these via per-language helpers
(`bb_f32_from_bits` in C/C++, `math.Float32frombits` in Go,
`Float.intBitsToFloat` in Java, `BitConverter.GetBytes` in C#,
`struct.pack` in Python, `Float32Array`/`BigUint64Array` in JS).

## Custom getter / setter (`GetterSetter` struct)

The `GetterSetter` scenario uses `Case.Accessors` (a list of
`AccessorRoundTrip{SetterName, SetArg, GetterName, Expected, Tol,
NoRoundTrip}`) instead of `Setup`/`Assert`. Each emitter has a dedicated
`emit*_getterSetter` path that calls the custom setter, optionally
encode/decodes, and asserts the getter result.

## Static loop-count checks (`-unroll`)

`compiler/test_options.sh` section 11b regenerates `testcase.bb` at
`-unroll=-1`, `-unroll=0`, `-unroll=4` and counts `for` loops in the output
to confirm the unroll threshold logic works. The runtime matrix in section
11 re-runs every language's full driver (which now includes the unroll-test
structs `SmallArrays` / `MediumArrays` / `LargeArrays` / `VeryLargeArrays`
/ `MixedArrays`) under unroll values `-1, 0, 1, 2, 4, 6, 8, 16, 32` so
encode/decode correctness is verified at every threshold.
